# Architecture

## Runtime boundaries

The Chrome extension is an untrusted remote client. Caddy terminates TLS and forwards requests to one API process. Web and extension sessions are persisted in SQLite. A worker on the same host claims durable analysis jobs in short write transactions and sends bounded requests to the configured Ollama origin.

```text
Google Meet DOM
  -> MV3 content script
  -> persistent chrome.storage.local upload queue
  -> HTTPS /api/v1/meetings/ingest
  -> authentication + membership policy + Zod validation
  -> SQLite transaction (meeting, segments, idempotency result)
  -> dashboard

POST /api/v1/meetings/:id/analyze
  -> analysis row + SQLite job
  -> serialized worker claim
  -> server-configured Ollama outside the transaction
  -> schema-validated JSON
  -> completed/failed analysis state
```

## SQLite design

SQLite is configured with foreign keys, WAL, `synchronous=NORMAL`, memory-backed temporary storage, and a 15-second default busy timeout. WAL lets readers continue during a writer. Write transactions remain intentionally short: job claims commit before Ollama calls, ingestion validates before opening its transaction, and expensive work occurs outside locks.

The supported topology is one host and one local SQLite volume. The API and worker can be separate processes on that host. Network filesystems, shared volumes across hosts, and horizontal API/worker replicas are unsupported. `OLLAMA_MAX_CONCURRENCY` defaults to one and is capped at four; increasing it does not increase SQLite write concurrency.

Transcript search uses an external-content FTS5 table maintained by insert/delete/update triggers. Tenant scope is still enforced through the meeting join; FTS never becomes an authorization boundary.

## Source layout

- `src/`: React dashboard and role-aware views.
- `server/routes/`: versioned HTTP boundaries.
- `server/auth/`: password, cookie session, extension token, CSRF, and policies.
- `server/services/`: tenant-scoped operations, audits, statistics, and jobs.
- `server/integrations/`: Ollama protocol and response validation.
- `server/db/`: Drizzle SQLite schema, client pragmas, and migrations.
- `packages/shared/`: roles, permissions, Zod contracts, and error types.
- `extension/`: MV3 capture, session settings, retry queue, and origin-aware build.

## Data and authorization

Users, workspaces, memberships, meetings, transcript segments, analyses, normalized decisions/action items, sessions, idempotency keys, jobs, and audit logs reside in SQLite. JSON fields have `json_valid` checks. Public IDs are UUID-shaped opaque strings generated by SQLite. Foreign keys and cascade/restrict rules are explicit. Meeting deletion is a deliberate soft delete.

The client-selected workspace is never trusted alone. Middleware verifies membership and permission. Extension workspace comes from its validated session. Meeting reads, changes, deletion, and analysis include both meeting ID and workspace ID. A known resource ID cannot cross a tenant boundary.

## Deployment modes

`DEPLOYMENT_MODE=local` permits localhost HTTP and development secrets. `DEPLOYMENT_MODE=server` requires production mode, HTTPS, non-loopback binding, exact CORS origins, strong unique secrets, constrained proxy trust, and a persistent `file:` database URL. There is no JSON-store production path.
